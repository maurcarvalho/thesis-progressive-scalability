This appendix provides definitions for key technologies, frameworks, and architectural patterns referenced throughout this thesis.

# Infrastructure and Data

+----------------+--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Term**       | Definition                                                                                                                                                                                                                                                                     |
+:===============+:===============================================================================================================================================================================================================================================================================+
| **Term**       | Definition                                                                                                                                                                                                                                                                     |
+----------------+--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| Apache Kafka   | A distributed event-streaming platform that provides high-throughput, fault-tolerant publish-subscribe messaging. In this work, Kafka serves as the event bus for inter-module communication, with per-module topics and consumer groups. Version: KRaft 3.7 (Zookeeper-free). |
+----------------+--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **PostgreSQL** | An open-source object-relational database system. In the proposed architecture, each module owns an isolated schema within a shared PostgreSQL instance, enforced through database roles that deny cross-schema access. Version: 16.x.                                         |
+----------------+--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Redis**      | An in-memory key-value data store used for caching and request deduplication. Cache patterns include product catalog (TTL 5 min), order status (write-through, TTL 1 hr), and idempotency keys (TTL 24 hr). Version: 7.2.                                                      |
+----------------+--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+

# Architectural Patterns

+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Term**                        | Definition                                                                                                                                                                                                                                                             |
+:================================+:=======================================================================================================================================================================================================================================================================+
| **Term**                        | Definition                                                                                                                                                                                                                                                             |
+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| Modular Monolith (Modulith)     | An architectural style in which a single deployable unit is decomposed into well-defined, loosely coupled modules with explicit boundaries. Modules communicate through in-process interfaces and share a deployment artifact but maintain independent data ownership. |
+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Saga Pattern**                | A sequence of local transactions coordinated to maintain data consistency across module boundaries without distributed locks. Each step has a compensating action that undoes its effect in case of downstream failure (backward recovery).                            |
+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Strangler Fig**               | A migration pattern in which new functionality is incrementally built alongside the legacy system, gradually routing traffic to the new implementation until the old code can be removed. Referenced in the context of progressive module extraction ($L_3$).          |
+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Anti-Corruption Layer (ACL)** | A translation boundary that prevents the domain model of one module from being corrupted by the models of adjacent modules or external systems. Implemented as versioned event contracts and DTOs at module boundaries.                                                |
+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Bounded Context**             | A Domain-Driven Design concept defining the boundary within which a particular domain model is consistent and valid. In the modulith architecture, each module corresponds to one bounded context with its own schema and ubiquitous language.                         |
+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Event Sourcing**              | A pattern in which state changes are persisted as an append-only sequence of domain events rather than mutable records. Referenced as a complementary technique for audit trails and temporal queries within individual modules.                                       |
+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **Change Data Capture (CDC)**   | A technique that detects and propagates data changes from a database to downstream consumers (e.g., via Debezium and Kafka Connect). Referenced as an alternative to application-level event publishing for $L_2$/$L_3$ data isolation.                                |
+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
| **CQRS**                        | Command Query Responsibility Segregation, a pattern that separates the write model (commands) from the read model (queries), allowing each to be optimized independently. Often paired with event sourcing in module boundaries.                                       |
+---------------------------------+------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------+
