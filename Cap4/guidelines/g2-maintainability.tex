\section{G2: Embed Maintainability}
\label{sec:g2-embed-maintainability}

G2 embeds maintainability as an architectural property by treating the cost of change as a design constraint. While G1 enforces explicit modular boundaries, G2 ensures those boundaries remain effective over time through stable contracts and continuous detection of architectural drift~\cite{gravanis2021dont,blinowski2022monolithic}.

\subsection*{Intent and Rationale}
The primary driver of maintainability degradation is not system size but the accumulation of hidden coupling, ambiguous dependency direction, and architectural drift~\cite{gravanis2021dont,blinowski2022monolithic}. G2 constrains how modules interact and how dependencies are introduced, prioritizing change locality and controlled dependency growth over short-term reuse so that the effort to evolve the system scales sublinearly with its size~\cite{arya2024beyond,berry2024isItWorth}.

\subsection*{Applicability}
G2 applies once G1 boundaries are in place and the system is expected to evolve continuously. It does not prescribe specific frameworks or CI pipelines; its scope is limited to metrics and design practices that influence long-term changeability.

\subsection*{Objectives}

\emph{Research contributions:}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Frame maintainability as bounded cost of change:} Limit change propagation so that most modifications remain confined to a single bounded context.
  \item \emph{Quantify boundary sustainability:} Establish metrics that distinguish point-in-time correctness (G1) from health over time, detecting drift before it becomes irreversible.
  \item \emph{Formalize the sustainability gap:} Demonstrate that a system can pass all G1 gates while accumulating maintainability debt through structurally valid patterns.
\end{itemize}

\emph{Implementation objectives:}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Stable dependency direction:} Prevent cycles and cascading change through disciplined layering and contract-first boundaries.
  \item \emph{Controlled API surface:} Constrain public surface growth to preserve refactoring freedom and minimize downstream obligations.
  \item \emph{Automated drift detection:} Track structural signals continuously so that degradation is caught while corrective actions remain localized.
  \item \emph{Independent module testability:} Ensure each module is testable using only its declared dependencies.
\end{itemize}

\subsection*{Key Principles}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Information hiding as the unit of modularity:} A module's boundary is defined by the design decisions it conceals, not by its size or feature count~\cite{parnas1972informationhiding}. Cross-module reuse of internal code is treated as a maintainability risk unless mediated through an explicit contract.
  \item \emph{Dependency direction reflects stability:} Dependencies flow toward stable abstractions; unstable modules never become dependencies of stable ones~\cite{martin2012clean}. Cycles violate this principle and are treated as maintainability violations.
  \item \emph{Change locality within bounded contexts:} Each bounded context owns its model and language; cross-context interaction occurs through explicit contracts, not shared internals~\cite{evans2003ddd}. Most changes should be implementable within a single module.
  \item \emph{Coupling is a spectrum, not a binary:} Data coupling through published events is preferable to content coupling through shared internals~\cite{yourdon1979structured}. G2 metrics quantify where each module sits on this spectrum, making drift visible before it becomes irreversible.
\end{itemize}

\subsection*{Implementation Mechanisms}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Contract-first module boundaries:} Modules interact exclusively through declared public interfaces or published domain events, never through internal types or shared database access~\cite{richardson2018microservices}.
  \item \emph{Single auditable entrypoint:} Each module exposes one public surface (e.g., \texttt{index.ts}) that defines its entire external contract. Internal code is unreachable from outside.
  \item \emph{Centralized composition root:} All cross-module wiring lives in one location, making coupling visible, auditable, and measurable~\cite{FordParsons2017}.
  \item \emph{Continuous fitness evaluation:} G2 metrics are recorded per sprint or release as architectural fitness functions~\cite{thoughtworks2017fitnessfunction}, detecting drift while corrective actions remain localized.
\end{itemize}

\subsection*{Common Failure Modes and Anti-Patterns}

G1 anti-patterns (convenience imports, implicit wiring, circular dependencies) remain relevant at the G2 level. The two patterns below pass G1 gates but erode maintainability over time:

\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \textbf{Overloaded Module APIs}: API surface grows, discipline ratio drops ($G_{\mathrm{api}} \uparrow$, $\rho_{\mathrm{api}} \downarrow$)\\
  A module's public entrypoint grows from 7 to 15 exports across three sprints. Each new export is a downstream contractual obligation: once another module depends on it, the symbol cannot be renamed or removed without coordinated changes~\cite{parnas1972informationhiding}.

  \item \textbf{Feature-Centric Coupling Across Modules}: co-change frequency rises ($CCI \uparrow$)\\
  Adding a cross-cutting feature (e.g., "gift wrapping") requires simultaneous changes to ordering (option flag), billing (fee calculation), and fulfillment (packaging instructions). The feature, not the module, becomes the unit of change, diluting ownership and increasing coordination cost~\cite{FordParsons2017}.
\end{itemize}

\subsubsection*{The Sustainability Gap}

\noindent
A system can pass every G1 gate and still accumulate maintainability debt through structurally valid but unsustainable patterns. Where G1's Enforcement Gap argues that architectural knowledge without automation is insufficient, G2's Sustainability Gap argues that point-in-time correctness without tracking over time is insufficient. A single snapshot reveals whether boundaries are intact; only a time series reveals whether they are drifting.

\subsection*{Core Contribution}

The contribution is not the individual practices (encapsulation, contract stability, dependency analysis are well-established) but a unified model that makes boundary sustainability measurable and trend-aware:

\begin{enumerate}[itemsep=8pt,topsep=2pt]
  \item \emph{The Sustainability Gap.} G2 formalizes the gap between boundary correctness (G1) and boundary sustainability: a system can pass all G1 gates while accumulating maintainability debt through structurally valid patterns.

  \item \emph{Coupling spectrum applied to modular monoliths.} Yourdon and Constantine's coupling types~\cite{yourdon1979structured} were defined for modules in general. Applying them to bounded contexts in a monolith, where no deployment boundary enforces separation, and measuring position on the spectrum through $\rho_{\mathrm{api}}$, $CCI$, and $\phi_m$, is a novel operationalization. The composite $\mu_m$ feeds directly into G3's extraction readiness score.

  \item \emph{Graduated degradation detection.} The exercises demonstrate that each failure mode produces a specific, measurable signal while passing all G1 boundary tests, validating the complementarity of G1 and G2.
\end{enumerate}

\subsection*{Literature Support Commentary}
Modular monolith architectures can achieve maintainability comparable to microservices when boundary discipline is sustained over time~\cite{gravanis2021dont,berry2024isItWorth,blinowski2022monolithic,wolfart2021modernizing}. Change coupling as a health signal has been explored in microservices migration contexts~\cite{fritzsch2019,deLauretis2019from}; G2 adapts this insight to monoliths, where the absence of deployment boundaries makes temporal coupling harder to detect. The evolutionary architecture perspective~\cite{FordParsons2017} and fitness functions~\cite{thoughtworks2017fitnessfunction} provide the philosophical foundation; G2 operationalizes it through concrete metrics grounded in G1's boundary model.

\subsection*{G2 Applied: Maintainability in the Tiny Store}
\label{sec:g2-applied}

G2 is demonstrated in the Tiny Store reference implementation.\footnote{Full source: \url{https://github.com/maurcarvalho/tiny-store}} Where G1 focused on the Orders--Inventory boundary, G2 shifts to Payments and Shipments, whose evolution patterns reveal maintainability risks invisible to boundary checks alone.

\subsection*{Tutorial: Step-by-Step Application}

\subsubsection*{Step 1: Extract the Dependency Direction Map}

\begin{lstlisting}[language=bash,caption={Extract and filter the module dependency graph},label={lst:g2-dep-graph}]
npx nx graph --file=dep-graph.json
\end{lstlisting}

Every bounded-context module depends only on shared libraries; no module-to-module edge exists:

\begin{lstlisting}[language=bash,caption={Expected dependency direction (baseline)},label={lst:g2-dep-baseline}]
  modules-orders    -> shared-domain, shared-infrastructure
  modules-inventory -> shared-domain, shared-infrastructure
  modules-payments  -> shared-domain, shared-infrastructure
  modules-shipments -> shared-domain, shared-infrastructure
\end{lstlisting}

\noindent
If a module-to-module edge appears, coupling has migrated into module internals.

\subsubsection*{Step 2: Measure the API Surface per Module}

\begin{lstlisting}[language=bash,caption={Count exported symbols per module},label={lst:g2-api-surface}]
for mod in orders inventory payments shipments; do
  echo "$mod: $(grep -c '^export' libs/modules/$mod/src/index.ts) exports"
done
# orders: 20, inventory: 17, payments: 8, shipments: 9
\end{lstlisting}

These numbers form the $t_0$ snapshot for $G_{\mathrm{api}}$.

\subsubsection*{Step 3: Compute $\rho_{\mathrm{api}}$}
All imports in the composition root (\texttt{register-listeners.ts}) use the public path \texttt{@tiny-store/modules-*}, so $\rho_{\mathrm{api}} = 1.0$. Any internal path import would lower this ratio.

\subsubsection*{Step 4: Detect Change Coupling}

\begin{lstlisting}[language=bash,caption={Detect co-changing modules in recent commits},label={lst:g2-cci}]
git log --oneline -50 --name-only | \
  awk '/^[a-f0-9]/{c=$1; next}
       /libs\/modules\//{split($0,a,"/"); m[c][a[3]]}
       END{for(c in m) if(length(m[c])>1) print c": "length(m[c])" modules"}'
\end{lstlisting}

In the baseline, only infrastructure commits touch multiple modules; no feature commit crosses module boundaries ($CCI = 0.0$ for feature work).

\subsection*{Exercise Walkthrough: Controlled Maintainability Degradation}
Each exercise introduces a change that passes all G1 boundary tests but degrades a specific G2 metric. Exercises use Payments and Shipments exclusively, with no overlap with G1's Orders--Inventory examples.

\subsubsection*{Exercise 1: Convenience Shortcut That Passes G1 ($\rho_{\mathrm{api}} \downarrow$, undeclared dep $\uparrow$)}

In \texttt{libs/modules/payments/src/index.ts}, re-export an internal enum:
\begin{lstlisting}[language=TypeScript,caption={Step 1: Payments exports an internal enum},label={lst:g2-ex1-step1}]
// "Temporary" export added for cross-module convenience
export * from './domain/enums/payment-status.enum';
\end{lstlisting}

\noindent
Then consume it in \texttt{libs/modules/shipments/src/features/create-shipment/service.ts}:
\begin{lstlisting}[language=TypeScript,caption={Step 2: Shipments depends on Payments' internal type},label={lst:g2-ex1-step2}]
import { PaymentStatus } from '@tiny-store/modules-payments';

if (paymentStatus !== PaymentStatus.SUCCEEDED) {
  throw new Error('Cannot ship unpaid order');
}
\end{lstlisting}

\needspace{4\baselineskip}
\begin{lstlisting}[language=bash,caption={Verify G1 passes but G2 degrades},label={lst:g2-ex1-run}]
npm run test:boundary          # G1: PASSES (public entrypoint)
npx nx graph --file=dep-graph.json
# New edge: modules-shipments -> modules-payments (did not exist at baseline)
\end{lstlisting}

\noindent
G1 passes, but the Nx dependency graph gains a new module-to-module edge. Shipments can no longer evolve independently of Payments' domain model.\\
\textbf{Fix:} Remove the enum export; enrich the \texttt{OrderPaid} event payload with payment status instead.

\subsubsection*{Exercise 2: Transitive Orchestration ($C_{\mathrm{forbid}} \uparrow$, blast radius)}

Modify the \texttt{OrderPaid} subscription in \texttt{register-listeners.ts} to query Payments before creating a shipment:

\begin{lstlisting}[language=TypeScript,caption={Violation: three-module orchestration in one handler},label={lst:g2-ex2-violation}]
import { GetPaymentHandler } from '@tiny-store/modules-payments';

const getPaymentHandler = new GetPaymentHandler(db);

eventBus.subscribe('OrderPaid', async (event) => {
  const { orderId } = event.payload;
  const payment = await getPaymentHandler.handle(orderId);
  if (payment.status === 'REFUNDED') return;
  const order = await getOrderHandler.handle(orderId);
  await createShipmentHandler.handle({
    orderId, shippingAddress: order.shippingAddress,
  });
});
\end{lstlisting}

\noindent
G1 passes, but the handler now orchestrates three modules in sequence. A change to any handler's return type expands the blast radius across Payments, Orders, and Shipments.\\
\textbf{Fix:} Enrich the \texttt{OrderPaid} event payload with all data Shipments needs. If orchestration is necessary, extract it into a dedicated saga.

\subsubsection*{Exercise 3: Public Surface Bloat ($G_{\mathrm{api}} \uparrow$)}

Add five valid handler exports to \texttt{libs/modules/payments/src/index.ts}:

\begin{lstlisting}[language=TypeScript,caption={Surface bloat: valid but excessive exports},label={lst:g2-ex3-violation}]
// --- NEW: 5 additional query handlers in one sprint ---
export * from './features/get-payments-by-order/handler';
export * from './features/get-payments-by-status/handler';
export * from './features/get-payments-by-date/handler';
export * from './features/refund-payment/handler';
export * from './features/retry-payment/handler';
\end{lstlisting}

\noindent
G1 passes ($C_{\mathrm{leak}} = 0$, all exports are handlers). But $|API_{\mathrm{payments}}|$ grew 63\% in one sprint (8$\to$13). Each export is a downstream contract that constrains future refactoring.\\
\textbf{Fix:} Remove exports consumed only within the module. Set a $G_{\mathrm{api}}$ threshold in CI.

\subsection*{Conclusion}

G1 ensures boundaries are \emph{correct} at any point in time; G2 ensures they remain \emph{sustainable in cost of change} over time. The exercises confirmed that each failure mode produces a measurable signal while passing all G1 gates, validating the complementarity of the two guidelines.

With boundaries enforced (G1) and maintainability preserved (G2), the architecture retains the option to evolve incrementally. G3 addresses the next concern: which scaling interventions are proportional to which bottlenecks, and in what order.
