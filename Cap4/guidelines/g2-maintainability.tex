\section{G2: Embed Maintainability}
\label{sec:g2-embed-maintainability}

G2 embeds maintainability as an architectural property by framing it economically: the architecture constrains the expected effort, risk, and coordination required to modify the system. G2 operationalizes this through explicit design constraints, stable contracts, and continuous detection of architectural drift.

G2 builds directly on G1. While G1 enforces the existence of explicit modular boundaries, G2 ensures that those boundaries remain effective over time. Together, these guidelines prevent the gradual erosion that often causes traditional monoliths to regress into tightly coupled systems.

\subsection*{Intent and Rationale}
Empirical studies consistently show that the primary driver of maintainability degradation is not system size, but the accumulation of hidden coupling, ambiguous dependency direction, and architectural drift~\cite{gravanis2021dont,blinowski2022monolithic}. G2 treats maintainability as a preventive concern: constraining how modules interact, how dependencies are introduced, and how responsibilities are distributed. The guideline prioritizes change locality, stable contracts, and controlled dependency growth over short-term reuse, ensuring that the effort to evolve the system scales sublinearly with its size~\cite{arya2024beyond,berry2024isItWorth}.

\subsection*{Conceptual Overview}
Maintainability is embedded by designing modules so that change remains local and predictable. Each bounded context exposes a narrow public surface representing stable capabilities rather than internal structure. Cross-module interaction occurs exclusively through explicit contracts (synchronous handler calls or asynchronous domain events). Dependency direction reflects responsibility and information flow, and architectural drift is detected through automated tracking of structural signals.

\subsection*{Applicability Conditions and Scope}
G2 applies to systems organized as modular monoliths, where multiple bounded context modules coexist within a single codebase and are expected to evolve continuously. The guideline assumes that:
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item Module boundaries are explicitly defined and at least partially enforced, as established by G1.
  \item Each module has clear ownership and a well-defined responsibility.
  \item Architectural drift is a realistic risk during ongoing feature development.
\end{itemize}

G2 does not prescribe specific frameworks, build tools, or CI pipelines. Its scope is limited to metrics definition and design-level practices that influence long-term changeability.

\subsection*{Objectives}

\emph{Research contributions:}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Frame maintainability as bounded cost of change:} Limit change propagation so that most modifications remain confined to a single bounded context.
  \item \emph{Quantify boundary sustainability:} Establish metrics that distinguish point-in-time correctness (G1) from longitudinal health, detecting drift before it becomes irreversible.
  \item \emph{Formalize the sustainability gap:} Demonstrate that a system can pass all G1 gates while accumulating maintainability debt through structurally valid patterns.
\end{itemize}

\emph{Implementation objectives:}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Stable dependency direction:} Prevent cycles and cascading change through disciplined layering and contract-first boundaries.
  \item \emph{Controlled API surface:} Constrain public surface growth to preserve refactoring freedom and minimize downstream obligations.
  \item \emph{Automated drift detection:} Track structural signals continuously so that degradation is caught while corrective actions remain localized.
  \item \emph{Independent module testability:} Ensure each module is testable using only its declared dependencies.
\end{itemize}

\subsection*{Key Principles}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Encapsulation before reuse:} Cross-module reuse of internal code is treated as a maintainability risk unless mediated through an explicit contract.
  \item \emph{Contract over structure:} Modules depend on declared interfaces or events, not on internal implementation details.
  \item \emph{Unidirectional dependency flow:} Dependency direction reflects responsibility and information flow, and cycles are considered maintainability violations.
  \item \emph{Change locality:} Most changes should be implementable within a single module.
  \item \emph{Evolution over speculation:} Abstractions are introduced to support observed change, not anticipated reuse.
\end{itemize}

\subsection*{Implementation Mechanisms}
G2 is implemented through established software engineering practices rather than framework-specific mechanisms:
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Layered internal structure:} Within each bounded context, code is organized into domain, application, and infrastructure concerns, ensuring that business rules remain insulated from volatile technical details.
  \item \emph{Explicit module entrypoints:} Each module exposes a single public surface that defines its externally visible capabilities.
  \item \emph{Contract-oriented collaboration:} Cross-module interaction occurs through synchronous interfaces or published domain events.
  \item \emph{Centralized composition:} Wiring between modules is visible and auditable, preventing implicit coupling.
  \item \emph{Governed exceptions:} Deviations from dependency or layering rules require explicit justification.
\end{itemize}

\subsection*{Common Failure Modes and Anti-Patterns}

\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \textbf{Cross-Module Internal Reuse}: encapsulation leakage rises, API discipline drops ($C_{\mathrm{leak}} \uparrow$, $\rho_{\mathrm{api}} \downarrow$)\\
  Reusing internal classes from another module couples the consumer to the provider's internal structure, forcing coordinated changes on every refactoring~\cite{parnas1972informationhiding}.

  \item \textbf{Implicit Dependency Introduction}: undeclared dependencies rise, isolation tests fail ($C_{\mathrm{undecl}} \uparrow$, $P_{\mathrm{iso}} \downarrow$)\\
  Dependencies introduced through reflection, configuration, or DI wiring without explicit declaration bypass architectural review~\cite{fairbanks2010justenough}.

  \item \textbf{Bidirectional or Cyclic Dependencies}: forbidden dependency count rises ($C_{\mathrm{forbid}} \uparrow$)\\
  Mutual module dependencies prevent independent evolution and are expensive to remove once established~\cite{martin2012clean,yourdon1979structured}.

  \item \textbf{Overloaded Module APIs}: API surface grows, discipline ratio drops ($G_{\mathrm{api}} \uparrow$, $\rho_{\mathrm{api}} \downarrow$)\\
  Uncontrolled growth of a module's public surface freezes internal structure and creates downstream contractual obligations~\cite{parnas1972informationhiding}.

  \item \textbf{Feature-Centric Coupling Across Modules}: co-change frequency rises ($CCI \uparrow$)\\
  When features rather than modules become the unit of change, coordination cost rises and accountability is diluted~\cite{FordParsons2017}.
\end{itemize}

\subsubsection*{The Sustainability Gap}

\noindent
The anti-patterns above share a common trait: they are invisible to boundary enforcement alone. A system can pass every G1 gate and still accumulate maintainability debt through structurally valid but economically unsustainable patterns. This gap between boundary correctness and boundary sustainability is the central challenge G2 addresses. Where G1's Enforcement Gap argues that architectural knowledge without automation is insufficient, G2's Sustainability Gap argues that point-in-time correctness without longitudinal tracking is insufficient. A single snapshot of the dependency graph reveals whether boundaries are intact; only a time series reveals whether they are drifting. The metrics introduced below operationalize this longitudinal perspective, transforming maintainability from an aspirational property into a measurable, trend-aware constraint.

\noindent
\emph{Formal definitions.} Let $M$ be the set of modules and $R$ the multiset of cross-module references.

\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \textbf{API-Only Dependency Ratio ($\rho_{\mathrm{api}}$).}
  Measures the proportion of cross-module references that target public entrypoints rather than internal paths.
  \[
    \rho_{\mathrm{api}} = \frac{|R_{\mathrm{api}}|}{|R|}
  \]
  where $R_{\mathrm{api}} \subseteq R$ = references targeting only the provider module's declared public surface.\\
  \emph{Target:} $\rho_{\mathrm{api}} \geq 0.95$~\cite{parnas1972informationhiding}.\\
  \emph{Note:} Boundary bypass mechanisms (reflection, dynamic loading, configuration-driven wiring) that circumvent static enforcement should be counted as non-API references when computing $\rho_{\mathrm{api}}$.

  \item \textbf{Public API Surface Growth Rate ($G_{\mathrm{api}}$).}
  Measures the rate at which a module's exported symbols increase, signaling accumulating downstream obligations.
  \[
    G_{\mathrm{api}}(m) = \frac{|API_m(t)| - |API_m(t - \Delta t)|}{\Delta t}
  \]
  where $API_m(t)$ = set of public symbols exported by module $m$ at time $t$, $\Delta t$ = measurement period (sprint or release).\\
  \emph{Target:} $G_{\mathrm{api}}(m) \to 0$~\cite{parnas1972informationhiding}.

  \item \textbf{Change Coupling Index ($CCI$).}
  Measures temporal coupling: how often modules change together in commits, revealing hidden dependencies invisible to static analysis.
  \[
    CCI = \frac{\sum_{A \neq B} C(A,B)}{\sum_{A} C(A)}
  \]
  where $C(A,B)$ = commits where modules $A$ and $B$ both change, $C(A)$ = total commits touching module $A$.\\
  \emph{Target:} $CCI \leq 0.2$~\cite{FordParsons2017}.

  \item \textbf{Module Complexity Concentration ($\Gamma$).}
  Measures whether one module dominates the system's public surface, becoming a bottleneck for cross-cutting changes.
  \[
    \Gamma = \frac{\max_{m \in M} |API_m|}{\frac{1}{|M|} \sum_{m \in M} |API_m|}
  \]
  where $|API_m|$ = number of public symbols exported by module $m$.\\
  \emph{Target:} $\Gamma \leq 3.0$~\cite{martin2012clean}.

  \item \textbf{Contract Stability Index ($\sigma_m$).}
  Measures the proportion of a module's public API that remains unchanged between releases, capturing contract churn invisible to growth rate.
  \[
    \sigma_m = \frac{|\operatorname{API}_m(t) \cap \operatorname{API}_m(t - \Delta t)|}{|\operatorname{API}_m(t - \Delta t)|}
  \]
  where $\operatorname{API}_m(t)$ and $\operatorname{API}_m(t - \Delta t)$ = public symbol sets at current and previous measurement.\\
  \emph{Target:} $\sigma_m \geq 0.9$ per release cycle~\cite{FordParsons2017}.

  \item \textbf{Fan-In Concentration ($\phi_m$).}
  Measures the fraction of modules that depend on module $m$; high fan-in for domain modules signals responsibilities expanding beyond the bounded context.
  \[
    \phi_m = \frac{|\{A \in M \setminus \{m\} : \exists\, r \in \operatorname{refs}(A \to m)\}|}{|M| - 1}
  \]
  where $\operatorname{refs}(A \to m)$ = set of references from module $A$ to module $m$.\\
  \emph{Target:} $\phi_m \leq 0.5$ for bounded context modules~\cite{yourdon1979structured}.

  \item \textbf{Maintainability Index ($\mu_m$).}
  Synthesizes G2's per-module signals into a single composite score, analogous to G1's boundary health ($b_m$) and G3's extraction readiness ($\varepsilon_m$)~\cite{FordParsons2017}.
  \[
    \mu_m = 0.25 \cdot \rho_{\mathrm{api}}^{(m)} + 0.20 \cdot \sigma_m + 0.20 \cdot (1 - \phi_m) + 0.15 \cdot (1 - CCI^{(m)}) + 0.10 \cdot \frac{1}{\Gamma} + 0.10 \cdot (1 - \hat{G}_{\mathrm{api}}^{(m)})
  \]
  where $\rho_{\mathrm{api}}^{(m)}$ = API discipline for module $m$, $\hat{G}_{\mathrm{api}}^{(m)}$ = normalized API growth (0 = stable, 1 = doubling per period), $CCI^{(m)}$ = change coupling involving module $m$. Weights reflect relative impact on cost of change; teams should calibrate them. Validation through practitioner interviews is outlined in Chapter~5.\\
  \emph{Target:} $\mu_m \geq 0.8$. The maintainability index feeds into G3's extraction readiness score as the $a_m$ component.
\end{itemize}

\medskip
\noindent
\textbf{Compliance summary.} G2 compliance holds when all inherited boundary metrics pass (G1) and:
\[
  \rho_{\mathrm{api}} \geq 0.95 \land P_{\mathrm{iso}} = 1.0 \land \Gamma \leq 3.0 \land \sigma_m \geq 0.9 \land \phi_m \leq 0.5 \land \mu_m \geq 0.8
\]

\subsection*{Documentation Guidelines}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Dependency Direction Map:}
  Maintain a visual or textual map of the allowed dependency direction between modules. This map complements G1's module descriptor by making the intended information flow explicit and reviewable during architectural discussions.

  \item \emph{Contract Change Log:}
  Whenever a module's public surface changes (new handler, modified event schema, deprecated endpoint), record the change with version metadata. This log supports backward-compatibility analysis and informs downstream consumers of evolving contracts.

  \item \emph{Architectural Drift Incidents:}
  When a metric threshold is crossed (e.g., undeclared dependencies appear or the API discipline ratio drops below target), record the incident, root cause, and resolution. This creates a traceable history of maintainability interventions.
\end{itemize}

\subsection*{Tooling Capabilities Checklist}
Any open-source or proprietary tool used to support maintainability should address:
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Dependency Graph Visualization:} Generate and display the module dependency graph, highlighting cycles, undeclared dependencies, and forbidden references.
  \item \emph{API Surface Analysis:} Track the size and growth rate of each module's public surface over time (API growth rate).
  \item \emph{Change Coupling Detection:} Identify modules that frequently change together across commits (change coupling index), signaling feature-centric coupling or blurred ownership.
  \item \emph{Longitudinal Metric Tracking:} Store and visualize G2 metrics over time, enabling trend analysis and early detection of architectural drift.
  \item \emph{Contract Compatibility Checking:} Verify that changes to a module's public surface do not break downstream consumers, ideally integrated into the CI pipeline.
\end{itemize}

\subsection*{Core Contribution}

The contribution is not the individual practices (encapsulation, contract stability, dependency analysis are well-established) but a unified model that makes boundary sustainability measurable and trend-aware:

\begin{enumerate}[itemsep=8pt,topsep=2pt]
  \item \emph{The Sustainability Gap.} G2 formalizes the gap between boundary correctness (G1) and boundary sustainability: a system can pass all G1 gates while accumulating maintainability debt through structurally valid patterns.

  \item \emph{Longitudinal metric framework.} Six G2-specific metrics and a composite Maintainability Index ($\mu_m$) operationalize maintainability as bounded cost of change, transforming it from an aspirational property into a measurable, trend-aware constraint. The composite feeds directly into G3's extraction readiness score.

  \item \emph{Graduated degradation detection.} The exercises demonstrate that each failure mode produces a specific, measurable signal while passing all G1 boundary tests, validating the complementarity of G1 and G2.
\end{enumerate}

\subsection*{Literature Support Commentary}
Although maintainability is a central topic in software engineering, much of the literature treats it either as an abstract principle or as a collection of localized code quality metrics. Recent empirical studies and systematic reviews emphasize that maintainability improvements in modular monoliths are primarily driven by boundary discipline and dependency management rather than deployment decomposition alone~\cite{gravanis2021dont,berry2024isItWorth,blinowski2022monolithic}.

The relationship between modularity and maintainability has deep roots in information hiding~\cite{parnas1972informationhiding}. Recent systematic analyses confirm that modular monolith architectures can achieve maintainability levels comparable to microservices when boundary discipline is sustained over time~\cite{wolfart2021modernizing,blinowski2022monolithic}. The critical differentiator is not the deployment topology but the rigor with which dependency direction and encapsulation are enforced as the codebase evolves~\cite{abgaz2023decomposition}.

Change coupling as an architectural health signal has been explored in microservices migration contexts~\cite{fritzsch2019,deLauretis2019from}. G2 adapts this insight to modular monoliths, where the absence of deployment boundaries makes temporal coupling harder to detect. The evolutionary architecture perspective~\cite{FordParsons2017} and the concept of architectural fitness functions~\cite{thoughtworks2017fitnessfunction} provide the philosophical foundation; G2 operationalizes it through concrete metrics grounded in G1's boundary enforcement model.

\subsection*{G2 Applied: Maintainability in the Tiny Store}
\label{sec:g2-applied}

The following sections demonstrate G2's maintainability infrastructure in the Tiny Store reference implementation.\footnote{Full source: \url{https://github.com/maurcarvalho/tiny-store}} Where G1's tutorial focused on the Orders--Inventory boundary, G2 shifts attention to Payments and Shipments, smaller modules whose evolution patterns reveal maintainability risks invisible to boundary checks alone. The exercises demonstrate failure modes that pass all G1 gates yet degrade G2's longitudinal metrics.

\subsection*{Tutorial: Step-by-Step Application}

\subsubsection*{Step 1: Extract the Dependency Direction Map}

\begin{lstlisting}[language=bash,caption={Extract the module dependency graph},label={lst:g2-dep-graph}]
# Generate the Nx dependency graph (opens in browser)
npx nx graph

# For CI or scripted analysis, export as JSON:
npx nx graph --file=dep-graph.json
cat dep-graph.json | python3 -c "
import json, sys
g = json.load(sys.stdin)['graph']
for src, data in g['dependencies'].items():
    if 'modules-' in src:
        for dep in data:
            tgt = dep['target']
            if 'modules-' in tgt or 'shared-' in tgt:
                print(f'  {src} -> {tgt}')
"
\end{lstlisting}

Every bounded-context module depends only on shared libraries; no module-to-module edge exists:

\begin{lstlisting}[language=bash,caption={Expected dependency direction (baseline)},label={lst:g2-dep-baseline}]
  modules-orders    -> shared-domain
  modules-orders    -> shared-infrastructure
  modules-inventory -> shared-domain
  modules-inventory -> shared-infrastructure
  modules-payments  -> shared-domain
  modules-payments  -> shared-infrastructure
  modules-shipments -> shared-domain
  modules-shipments -> shared-infrastructure
\end{lstlisting}

\noindent
Cross-module coupling exists only in the composition root (\texttt{register-listeners.ts}), not inside any module. If a module-to-module edge appears in the graph, coupling has migrated into module internals.

\subsubsection*{Step 2: Measure the API Surface per Module}

\begin{lstlisting}[language=bash,caption={Count exported symbols per module},label={lst:g2-api-surface}]
for mod in orders inventory payments shipments; do
  count=$(grep -c "^export" \
    libs/modules/$mod/src/index.ts 2>/dev/null || echo 0)
  echo "$mod: $count exports"
done
\end{lstlisting}

Expected baseline output:

\begin{lstlisting}[language=bash,caption={API surface baseline},label={lst:g2-api-baseline}]
orders:    20 exports  (4 handlers + 4 DTOs + 7 events + 5 listeners)
inventory: 16 exports  (5 handlers + 5 DTOs + 3 events + 3 listeners)
payments:   7 exports  (2 handlers + 2 DTOs + 2 events + 1 listener)
shipments:  8 exports  (2 handlers + 2 DTOs + 3 events + 1 listener)
\end{lstlisting}

These numbers form the $t_0$ snapshot for $G_{\mathrm{api}}$. Payments (7) and Shipments (8) are lean contracts; G2's objective is to keep them that way.

\subsubsection*{Step 3: Audit Wiring Density and Compute $\rho_{\mathrm{api}}$}
G2 treats \texttt{register-listeners.ts} as a \emph{coupling density indicator}: subscription count reflects coordination cost, and its growth rate signals feature-centric coupling.

\begin{lstlisting}[language=bash,caption={Count cross-module subscriptions},label={lst:g2-wiring-density}]
grep -c "eventBus.subscribe" \
  apps/api/src/app/lib/register-listeners.ts
# Baseline: 23 subscriptions (13 event-store + 10 listener wiring)
\end{lstlisting}

\noindent
The API discipline ratio ($\rho_{\mathrm{api}}$) measures whether cross-module references target public entrypoints or internal paths. In Tiny Store, all imports in \texttt{register-listeners.ts} use the public path \texttt{@tiny-store/modules-*}:

\begin{lstlisting}[language=TypeScript,caption={All composition root imports use public entrypoints},label={lst:g2-rho-api}]
import { OrderPlacedListener } from '@tiny-store/modules-inventory';
import { InventoryReservedListener } from '@tiny-store/modules-orders';
import { PaymentProcessedListener } from '@tiny-store/modules-orders';
import { OrderConfirmedListener } from '@tiny-store/modules-payments';
import { OrderPaidListener } from '@tiny-store/modules-shipments';
// ... 8 more listener/handler imports from other modules

// None of these imports target internal paths 
// e.g., @tiny-store/modules-payments/src/domain/...
\end{lstlisting}

With all cross-module references routing through public surfaces, $\rho_{\mathrm{api}} = 1.0$. Any deviation means an internal path has leaked into the composition root.

\subsubsection*{Step 4: Detect Change Coupling via Co-Change Analysis}
The Change Coupling Index ($CCI$) captures modules changing together across commits, revealing hidden dependencies invisible to static analysis:

\begin{lstlisting}[language=bash,caption={Detect co-changing modules in recent commits},label={lst:g2-cci}]
# List modules touched per commit, flag co-changes
git log --oneline -50 --name-only | \
  awk '/^[a-f0-9]/{commit=$1; next}
       /^libs\/modules\//{split($0,a,"/"); mods[commit][a[3]]}
       END{for(c in mods) if(length(mods[c])>1)
         print c, ":", length(mods[c]), "modules"}'
\end{lstlisting}

In the Tiny Store baseline, the only multi-module commits are infrastructure changes (boundary refactoring, test additions) that intentionally touched all four modules simultaneously. No feature commit crosses module boundaries:

\begin{lstlisting}[language=bash,caption={Co-change analysis output (baseline)},label={lst:g2-cci-output}]
# Infrastructure commits (expected multi-module):
62354e5  2 modules: payments, shipments    # test suites added
2802a45  4 modules: inventory, orders,     # boundary refactoring
         payments, shipments
# Feature commits: none cross module boundaries
# CCI = 0.0 for feature work (healthy baseline)
\end{lstlisting}

In a healthy codebase, most feature commits touch a single module. When Payments and Shipments consistently change together, it indicates blurred responsibility boundaries, even without import violations.

\subsection*{Exercise Walkthrough: Controlled Maintainability Degradation}
Each exercise introduces a change that passes all G1 boundary tests but degrades a specific G2 metric. Exercises use Payments and Shipments exclusively (no overlap with G1's Orders--Inventory examples).

\subsubsection*{Exercise 0: Establish the G2 Maintainability Baseline}

\begin{lstlisting}[language=bash,caption={G2 maintainability baseline},label={lst:g2-ex0}]
# 1. Confirm G1 gates still pass
npm run test:boundary && npm run test:integration

# 2. Record API surface sizes
for mod in orders inventory payments shipments; do
  echo "$mod: $(grep -c '^export' \
    libs/modules/$mod/src/index.ts) exports"
done
# payments: 7, shipments: 8

# 3. Record rho_api
grep -c "@tiny-store/modules-" \
  apps/api/src/app/lib/register-listeners.ts
# All via public entrypoints -> rho_api = 1.0

# 4. Record dependency graph (no module-to-module edges)
npx nx graph --file=dep-graph.json
\end{lstlisting}

Baseline: $|API_{\mathrm{payments}}| = 7$, $|API_{\mathrm{shipments}}| = 8$, $\rho_{\mathrm{api}} = 1.0$, $C_{\mathrm{forbid}} = 0$, zero module-to-module edges.

\subsubsection*{Exercise 1: Convenience Shortcut That Passes G1 (API discipline ratio $\downarrow$, undeclared dependencies $\uparrow$)}

In \texttt{libs/modules/payments/src/index.ts}, re-export an internal enum:
\begin{lstlisting}[language=TypeScript,caption={Step 1: Payments exports an internal enum},label={lst:g2-ex1-step1}]
// "Temporary" export added for cross-module convenience
export * from './domain/enums/payment-status.enum';
\end{lstlisting}

\noindent
Then consume it in \texttt{libs/modules/shipments/src/features/create-shipment/service.ts}:
\begin{lstlisting}[language=TypeScript,caption={Step 2: Shipments depends on Payments' internal type},label={lst:g2-ex1-step2}]
import { PaymentStatus } from '@tiny-store/modules-payments';

// Now create-shipment logic checks payment status directly
if (paymentStatus !== PaymentStatus.SUCCEEDED) {
  throw new Error('Cannot ship unpaid order');
}
\end{lstlisting}

\needspace{4\baselineskip}
\begin{lstlisting}[language=bash,caption={Verify G1 passes but G2 degrades},label={lst:g2-ex1-run}]
npm run test:boundary          # G1: PASSES (public entrypoint)
grep -c "^export" libs/modules/payments/src/index.ts  # Was 7, now 8
npx nx graph --file=dep-graph.json
# New edge: modules-shipments -> modules-payments (did not exist at baseline)
\end{lstlisting}

\noindent
G1 passes (import uses public entrypoint), but the Nx dependency graph gains a new \texttt{modules-shipments $\rightarrow$ modules-payments} edge. Shipments can no longer evolve independently of Payments' domain model.\\
\textbf{Fix:} Remove the enum export; enrich the \texttt{OrderPaid} event payload with payment status instead.

\subsubsection*{Exercise 2: Transitive Orchestration in the Composition Root (forbidden dependencies $\uparrow$, blast radius)}

Modify the \texttt{OrderPaid} subscription in \texttt{register-listeners.ts} to query Payments before creating a shipment:

\begin{lstlisting}[language=TypeScript,caption={Violation: three-module orchestration in one handler},label={lst:g2-ex2-violation}]
import { GetPaymentHandler } from '@tiny-store/modules-payments';

// In registerListeners():
const getPaymentHandler = new GetPaymentHandler(db);

eventBus.subscribe('OrderPaid', async (event) => {
  const { orderId } = event.payload;
  // Query Payments to verify payment is not refunded
  const payment = await getPaymentHandler.handle(orderId);
  if (payment.status === 'REFUNDED') return;
  // Then query Orders for shipping address
  const order = await getOrderHandler.handle(orderId);
  await createShipmentHandler.handle({
    orderId,
    shippingAddress: order.shippingAddress,
  });
});
\end{lstlisting}

\needspace{4\baselineskip}
\begin{lstlisting}[language=bash,caption={Detect coupling density increase},label={lst:g2-ex2-run}]
npm run test:boundary          # G1: PASSES
grep -c "new.*Handler\|new.*Listener" \
  apps/api/src/app/lib/register-listeners.ts  # Was 12, now 13
# Chain: OrderPaid -> Payments -> Orders -> Shipments (3 modules)
\end{lstlisting}

\noindent
G1 passes, but the handler now orchestrates three modules in sequence. A change to any handler's return type breaks the composition root, expanding the blast radius across Payments, Orders, and Shipments simultaneously.\\
\textbf{Fix:} Enrich the \texttt{OrderPaid} event payload with all data Shipments needs. If orchestration is necessary, extract it into a dedicated saga with explicit ownership.

\subsection*{Conclusion}

G2 demonstrated that maintainability in a modular monolith cannot be reduced to boundary enforcement alone. G1 ensures boundaries are \emph{correct} at any point in time; G2 ensures they remain \emph{economically sustainable} over time. The exercises confirmed that each failure mode produces a specific, measurable signal while passing all G1 gates, validating the complementarity of the two guidelines.

With boundaries enforced (G1) and maintainability preserved (G2), the architecture retains the option to evolve incrementally while keeping the cost of change bounded. G3 addresses the next concern: which scaling interventions are proportional to which bottlenecks, and in what order.
