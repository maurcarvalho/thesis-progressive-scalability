\section{G1: Enforce Modular Boundaries}
\label{sec:g1-enforce-modular-boundaries}

G1 enforces clear modular boundaries to prevent unintended coupling, preserve encapsulation, and ensure that dependencies between modules remain explicit and verifiable through build- and test-time checks.

\subsection*{Intent and Rationale}
Modularity in a monolith should be treated as an enforceable architectural property rather than an assumed outcome of refactoring. Without enforcement, boundaries degrade through convenience-driven imports, shared utilities, and implicit wiring~\cite{parnas1972informationhiding}. G1 treats boundary enforcement as a preventive control: making coupling explicit, reviewable, and auditable.

\subsection*{Conceptual Overview}
Define clear boundaries around each bounded-context module so that:
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item Each module encapsulates its internal implementation and feature-level organization.
  \item Dependencies between modules occur only through explicitly declared and verifiable interfaces, whether synchronous (public API calls) or asynchronous (event subscriptions and command channels).
  \item Unintended coupling is prevented through automated enforcement, enabling independent evolution of modules within a single deployable system.
\end{itemize}


\subsection*{Applicability Conditions and Scope}
G1 is applicable when a system is decomposed into logical modules within a single codebase. Inter-module dependencies must be explicitly declared and automatically verifiable, independent of packaging, deployment, or distribution decisions. This guideline assumes a two-level decomposition strategy:
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Bounded Context Modules:} The codebase is decomposed into bounded-context modules (for example, under a \texttt{modules/} root), and each bounded context is treated as a first-class module boundary.
  \item \emph{Package-by-Feature Within Modules:} Inside each module, code is organized by feature (use cases, domain concepts, aggregates) to localize behavior and reduce scattering, without weakening the outer module boundary.
\end{itemize}
G1 focuses on enforcing the \emph{outer} boundary between bounded-context modules. The internal package-by-feature organization is treated as a complementary structuring mechanism, and its detailed rules are addressed in subsequent topics.

\subsection*{Objectives}

\emph{Research contributions:}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Operationalize modularity as enforceable property:} Define boundary enforcement as a measurable, automated constraint rather than an informal convention.
  \item \emph{Formalize the enforcement gap:} Demonstrate that conceptual knowledge of anti-patterns is insufficient without executable checks in the CI pipeline.
  \item \emph{Provide a composite boundary health metric:} Synthesize five boundary signals into a per-module score ($b_m$) that feeds G2 maintainability and G3 extraction readiness.
\end{itemize}

\emph{Implementation objectives:}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Encapsulation and controlled dependency:} Keep internal classes hidden; allow cross-module access only through declared interfaces.
  \item \emph{Automated early verification:} Detect boundary violations at build or test time, not at runtime.
  \item \emph{Architectural drift prevention:} Defend module boundaries against convenience-driven erosion patterns.
  \item \emph{Isolation guarantees:} Forbid specific module pairs from depending on each other.
\end{itemize}

\subsection*{Key Principles}
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Provided Interface vs.\ Internal Components:}
    \begin{itemize}[noitemsep,topsep=1pt,leftmargin=1.5em]
      \item \emph{Provided Interface:} Public classes, services, APIs, or events explicitly exposed to other modules.
      \item \emph{Internal Components:} Classes, utilities, or data that should remain private within the module.
    \end{itemize}

  \item \emph{Explicit Declarations:}
    All dependencies on other modules must be declared in a central descriptor to provide auto-verifiability (for example, annotation, configuration file, or YAML).

  \item \emph{Forbidden Dependencies:}
    Declare modules that must never depend on each other to enforce stronger isolation where necessary.

  \item \emph{Inter-Module Communication:}
    Use either synchronous calls via public interfaces or asynchronous event-driven channels, and avoid direct references to another module's internals.

  \item \emph{Boundary-First Discipline:}
    Package-by-feature improves cohesion within a bounded context, but it does not replace module boundaries. A feature slice must not become a cross-module dependency mechanism (for example, by importing internal feature code from other modules).
\end{itemize}

\subsection*{Implementation Mechanisms}
This research proposes implementing G1 through a combination of (i) structural encapsulation, (ii) explicit dependency declaration, and (iii) automated verification:
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Structural encapsulation:} Separate public API packages from internal packages, and restrict visibility so that only the declared API is importable across module boundaries.
  \item \emph{Declarative dependency model:} Maintain a module descriptor that defines which modules are required and which are forbidden, and keep it version-controlled with boundary changes.
  \item \emph{Automated verification:} Fail the build when undeclared cross-module references are detected, and optionally verify event subscriptions and command usage against declared dependencies.
\end{itemize}

\subsection*{Common Failure Modes and Anti-Patterns}

\subsubsection*{Structural Anti-Patterns}

\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \textbf{Convenience-Driven Imports}: encapsulation leakage rises ($C_{\mathrm{leak}} \uparrow$)\\
  A developer imports an internal entity class directly from another module because the path is shorter than going through the public API. Over time, dozens of such shortcuts create a web of hidden coupling that locks the provider's internal structure in place~\cite{parnas1972informationhiding}.

  \item \textbf{Shared Utility Creep}: undeclared dependencies rise ($C_{\mathrm{undecl}} \uparrow$)\\
  A \texttt{shared/utils} package starts with generic helpers (date formatting, string validation) but gradually absorbs domain logic such as \texttt{calculateShippingCost} or \texttt{validatePaymentStatus}. Modules couple through this shared layer rather than through explicit contracts~\cite{martin2012clean}.

  \item \textbf{Implicit Wiring via Framework Magic}: isolation tests fail ($P_{\mathrm{iso}} \downarrow$)\\
  DI containers, auto-configuration, or classpath scanning silently wire modules together. A \texttt{@ComponentScan} annotation may inject a bean from one module into another without any declared dependency, creating coupling invisible to static analysis~\cite{rocha2021catalogingDI}.

  \item \textbf{Event Handlers Outside the Composition Root}: event violations rise ($C_{\mathrm{event}} \uparrow$)\\
  Event subscriptions are scattered across module initializers rather than centralized in a single wiring location. Each hidden subscription is a cross-module dependency that works at runtime but is invisible to architectural review~\cite{richardson2018microservices}.

  \item \textbf{Boundary Enforcement Treated as Optional}: all boundary metrics degrade\\
  When boundary verification is not a mandatory CI gate, violations accumulate silently across sprints. Teams discover the erosion only when an extraction attempt reveals that modules are far more coupled than the architecture documentation suggests~\cite{thoughtworks2017fitnessfunction}.
\end{itemize}

\subsubsection*{Design-Level Anti-Patterns}

\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \textbf{Extreme Granularity}: undeclared dependencies across many small modules ($C_{\mathrm{undecl}} \uparrow$)\\
  Teams slice the system into excessively narrow domains such as \texttt{StockCheck}, \texttt{ReservationHold}, \texttt{WarehouseAllocation}, and \texttt{StockReplenishment} instead of grouping them under a cohesive bounded context like \texttt{Inventory}. Each micro-domain cannot fulfill a business workflow independently, forcing constant cross-module calls for routine operations~\cite{dragoni2017microservices,evans2003ddd}.

  \item \textbf{Speculative Generality}: unused interfaces leak across boundaries ($C_{\mathrm{leak}} \uparrow$)\\
  The codebase accumulates abstract factories, strategy patterns, and extension hooks designed for hypothetical requirements that may never materialize. Each speculative abstraction adds exported symbols that cross module boundaries without serving a current need. The YAGNI principle~\cite{extremeProgramming} applies at the architectural level.

  \item \textbf{Leaky Abstractions in Public Contracts}: encapsulation leakage rises ($C_{\mathrm{leak}} \uparrow$)\\
  A module exposes DTOs or event payloads that mirror its internal database schema (column names like \texttt{payment\_status\_id}, nullable fields reflecting storage trade-offs) rather than domain concepts. When the module refactors its schema, all consumers break even though no business rule changed~\cite{fowler2002,evans2003ddd}.

  \item \textbf{Circular Dependencies}: forbidden dependency count rises ($C_{\mathrm{forbid}} \uparrow$)\\
  Module A depends on B which depends on A, often through indirect paths spanning three or more modules. Cycles make independent testing and extraction impossible because changing any module in the cycle requires understanding all of them~\cite{martin2012clean,yourdon1979structured}. G1 treats any cycle as a blocking violation.

  \item \textbf{Test Coupling Across Boundaries}: isolation tests fail on cross-module fixtures ($P_{\mathrm{iso}} \downarrow$)\\
  Test code in one module directly instantiates domain objects from another module to build fixtures or assert on internal state. A refactoring of the provider's entities breaks the consumer's test suite, even though production code respects the boundary~\cite{fairbanks2010justenough}.
\end{itemize}

\noindent
The SOLID principles, properly applied, address many of these failure modes: Single Responsibility prevents god modules, Open-Closed means extension through published contracts rather than speculative abstractions, and Dependency Inversion ensures modules depend on stable interfaces~\cite{martin2012clean}. In a modular monolith, SOLID serves boundary discipline, not architectural ornamentation.

\subsubsection*{The Enforcement Gap}

\noindent
Recognizing anti-patterns and understanding SOLID principles is necessary but not sufficient. Most of the failure modes described above are well known by the software industry; experienced engineers recognize them on sight during code reviews and design discussions. Yet recognition alone does not prevent them. The gap between conceptual knowledge and daily practice persists because these principles are rarely operationalized as measurable, automated constraints~\cite{fairbanks2010justenough}. They exist as shared understanding in the team's collective memory, not as executable checks in the CI pipeline.

This gap compounds with growth. A single convenience import is harmless; hundreds of locally reasonable shortcuts across months of fast-paced delivery by multiple engineers and teams produce systemic architectural drift that no amount of documentation or periodic review can reverse. By the time the drift becomes visible, typically during an extraction attempt or a major refactoring, the cost of correction dwarfs the cost of prevention.

The metrics and enforcement gates introduced in this guideline (and extended throughout G2 through G6) represent a deliberate attempt to close this gap: translating what engineers already know into what the build system can verify on every commit.

\medskip
\noindent
\emph{Formal definitions.} In prose, metrics are referred to by name; the notation below supports unambiguous tooling implementation. Let $M$ be the set of bounded-context modules, $R$ the multiset of observed cross-module references (static imports, DI wiring, event handler registrations), $\operatorname{refs}(A \to B) \subseteq R$ the references from $A$ targeting symbols in $B$, $D \subseteq M \times M$ the declared allowed dependencies, $F \subseteq M \times M$ the explicitly forbidden dependencies, and $\operatorname{pub}(B)$ the set of symbols in module $B$'s declared public API surface.

\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \textbf{Undeclared Dependency Reference Count ($C_{\mathrm{undecl}}$).}
  Measures invisible coupling: references that exist in code but are not subject to architectural review~\cite{deLauretis2019from}.
  \[
    C_{\mathrm{undecl}} = \sum_{A \neq B} \left|\left\{ r \in \operatorname{refs}(A \to B) : (A, B) \notin D \right\}\right|
  \]
  where $D$ = set of declared allowed dependencies from the module descriptor.\\
  \emph{Target:} $C_{\mathrm{undecl}} = 0$.

  \item \textbf{Forbidden Dependency Reference Count ($C_{\mathrm{forbid}}$).}
  Measures known architectural violations where explicitly prohibited module pairs have references~\cite{martin2012clean}.
  \[
    C_{\mathrm{forbid}} = \sum_{A \neq B} \left|\left\{ r \in \operatorname{refs}(A \to B) : (A, B) \in F \right\}\right|
  \]
  where $F$ = set of explicitly forbidden dependencies.\\
  \emph{Target:} $C_{\mathrm{forbid}} = 0$.

  \item \textbf{Encapsulation Leakage Count ($C_{\mathrm{leak}}$).}
  Measures references that bypass the provider module's public API surface, coupling consumers to implementation details~\cite{parnas1972informationhiding}.
  \[
    C_{\mathrm{leak}} = \sum_{A \neq B} \left|\left\{ r \in \operatorname{refs}(A \to B) : \operatorname{target}(r) \notin \operatorname{pub}(B) \right\}\right|
  \]
  where $\operatorname{pub}(B)$ = set of symbols in module $B$'s declared public API surface.\\
  \emph{Target:} $C_{\mathrm{leak}} = 0$.

  \item \textbf{Module Isolation Test Pass Rate ($P_{\mathrm{iso}}$).}
  Measures hidden runtime dependencies invisible to static import analysis, by executing module tests with only declared dependencies available~\cite{fairbanks2010justenough}.
  \[
    P_{\mathrm{iso}} = \frac{|T_{\mathrm{pass}}|}{|T_{\mathrm{iso}}|}
  \]
  where $T_{\mathrm{iso}}$ = set of module-scoped isolation tests, $T_{\mathrm{pass}} \subseteq T_{\mathrm{iso}}$ = subset that passes.\\
  \emph{Target:} $P_{\mathrm{iso}} = 100\%$.

  \item \textbf{Event Subscription Boundary Violations ($C_{\mathrm{event}}$).}
  Measures coupling through data flow rather than code structure: event handlers registered without a declared dependency on the publisher module~\cite{richardson2018microservices}.
  \[
    C_{\mathrm{event}} = \left|\left\{ (A, B) \in H : (A, B) \notin D \right\}\right|
  \]
  where $H$ = set of $(subscriber, publisher)$ pairs derived from event handler registrations.\\
  \emph{Target:} $C_{\mathrm{event}} = 0$.

  \item \textbf{Boundary Health Score ($b_m$).}
  Synthesizes G1's per-module signals into a single composite score, analogous to G2's maintainability index ($\mu_m$) and G3's extraction readiness ($\varepsilon_m$)~\cite{FordParsons2017}.
  \[
    b_m = \begin{cases}
      1.0 & \text{if } C_{\mathrm{undecl}}^{(m)} = 0 \wedge C_{\mathrm{forbid}}^{(m)} = 0 \wedge C_{\mathrm{leak}}^{(m)} = 0 \wedge C_{\mathrm{event}}^{(m)} = 0 \wedge P_{\mathrm{iso}}^{(m)} = 1.0 \\
      1.0 - \frac{C_{\mathrm{undecl}}^{(m)} + C_{\mathrm{forbid}}^{(m)} + C_{\mathrm{leak}}^{(m)} + C_{\mathrm{event}}^{(m)}}{|R^{(m)}|} \cdot (1 - P_{\mathrm{iso}}^{(m)}) & \text{otherwise}
    \end{cases}
  \]
  where $C_{\bullet}^{(m)}$ = violation counts scoped to module $m$, $R^{(m)}$ = total cross-module references involving $m$, $P_{\mathrm{iso}}^{(m)}$ = isolation pass rate for $m$. The boundary health score feeds into G3's extraction readiness formula as the $b_m$ component.\\
  \emph{Target:} $b_m \geq 0.9$; a fully compliant module scores $b_m = 1.0$.
\end{itemize}

\medskip
\noindent
\textbf{Compliance summary.} G1 boundary compliance holds when:
\[
  C_{\mathrm{undecl}} = 0 \;\wedge\; C_{\mathrm{forbid}} = 0 \;\wedge\; C_{\mathrm{leak}} = 0 \;\wedge\; P_{\mathrm{iso}} = 1.0 \;\wedge\; C_{\mathrm{event}} = 0
\]

\subsection*{Tooling Capabilities Checklist}
Any open-source or proprietary tool used to enforce modular boundaries should support:
\begin{itemize}[itemsep=8pt,topsep=2pt]
  \item \emph{Module Discovery:} Automatically identify modules via conventions (for example, module folders, annotations) or explicit configuration.
  \item \emph{Boundary Verification:} Perform automated checks during build/test to detect undeclared cross-module references and fail on violations.
  \item \emph{Event-Driven Enforcement:} Track published events and subscribers, enforcing that only modules with declared subscriptions handle specific events.
  \item \emph{Isolated Testing:} Allow tests to load only a given module (and its declared dependencies), failing early if the module tries to wire code from undeclared modules.
  \item \emph{Documentation Artifacts:} Generate up-to-date artifacts or reports for inclusion in architecture documentation or CI/CD feedback.
  \item \emph{Runtime Validation (Optional):} Optionally perform runtime checks to catch dynamic boundary violations (for example, reflection) that compile-time checks might miss.
\end{itemize}

\subsection*{Core Contribution}

The contribution is not the individual practices (encapsulation, dependency declaration, automated testing are well-established) but a unified enforcement model that makes boundary discipline measurable and composable:

\begin{enumerate}[itemsep=8pt,topsep=2pt]
  \item \emph{The Enforcement Gap.} G1 formalizes the gap between conceptual knowledge of anti-patterns and executable enforcement: recognition without automation is insufficient under delivery pressure.

  \item \emph{Five-metric boundary model.} Undeclared dependencies, forbidden references, encapsulation leakage, isolation pass rate, and event subscription violations provide complete coverage of structural, runtime, and data-flow coupling.

  \item \emph{Composite boundary health score ($b_m$).} A per-module score that feeds directly into G2's maintainability index and G3's extraction readiness, creating a composable metric chain across guidelines.
\end{enumerate}

\subsection*{Literature Support Commentary}
The foundational argument for modular boundaries rests on information hiding~\cite{parnas1972informationhiding}: modules must encapsulate implementation details behind stable interfaces. The SLR conducted in Chapter~3 reveals that modularity is widely acknowledged as a prerequisite for sustainable architecture, yet few works operationalize it as an enforceable property. A survey of 63 decomposition studies shows that most approaches focus on \emph{identifying} boundaries but stop short of prescribing enforcement mechanisms~\cite{abgaz2023decomposition}.

Converging evidence supports G1's approach. The \textit{sliceable monolith} demonstrates that clean boundaries enable progressive decomposition without rewrites~\cite{montesi2021sliceable}. Component boundaries enforced at the programming model level, with deployment topology deferred, reinforce modularity as a build-time property~\cite{ghemawat2023towards}. Systematic analysis identifies boundary definition and dependency management as top decision drivers for modulith adoption~\cite{prakash2024systematic}.

Modernization roadmaps position coupling analysis as the first diagnostic step~\cite{wolfart2021modernizing}. Migration failures confirm that undeclared dependencies create invisible coupling that only surfaces under separation pressure~\cite{deLauretis2019from}. The modular monolith is increasingly recognized as a deliberate architectural choice rather than a transitional compromise~\cite{su2024modular,gravanis2021dont}. Industry experience at scale validates that automated boundary enforcement is the decisive factor in controlling complexity~\cite{shopify2020monolith,jovanovic2024rewriting}.

\section{G1 Applied: The Modular Tiny Store Example}
\label{sec:g1-demo-tiny-store}

The following sections operationalize G1 using Tiny Store, a small modular monolith maintained as an executable reference for this dissertation.\footnote{Full source: \url{https://github.com/maurcarvalho/tiny-store}} Two complementary feedback loops are used throughout: \texttt{npm run test:boundary} (structural rules) and \texttt{npm run test:integration} (cross-module behavior). The tutorial takes approximately 20--30 minutes.

\subsection*{Repository Orientation}
Tiny Store is an Nx monorepo with deployable applications under \texttt{apps/} and reusable libraries under \texttt{libs/}. Four bounded contexts (\texttt{orders}, \texttt{inventory}, \texttt{payments}, \texttt{shipments}) live under \texttt{libs/modules/}, each structured around \texttt{domain/} (aggregates, events, repositories), \texttt{features/} (use cases), and \texttt{listeners/} (event reactions). Each module exposes a public entrypoint (\texttt{src/index.ts}) that re-exports only approved symbols.

The HTTP boundary is \texttt{apps/api/} (Next.js App Router), which also hosts the composition root: \texttt{register-listeners.ts} centralizes all cross-module event subscriptions, making integration auditable. Shared code under \texttt{libs/shared/} is constrained to domain-neutral primitives (\texttt{shared-domain}) and infrastructure utilities (\texttt{shared-infrastructure}).

Cross-module collaboration follows an event-driven choreography: Orders emits domain events, Inventory reacts to reserve stock, and subsequent events drive Payments and Shipments.

\subsection*{Tutorial: Step-by-Step Application}

\begin{itemize}[itemsep=8pt,topsep=4pt]

  \item \emph{Step 0: Establish a Clean Baseline.}

  \begin{lstlisting}[language=bash,caption={Baseline checks for G1},label={lst:g1-baseline}]
npm install && npm test
npm run test:boundary
npm run test:integration
  \end{lstlisting}

  \item \emph{Step 1: Separate Public Surface from Internal Components.}
  Cross-module usage occurs only through module entrypoints (\texttt{@tiny-store/modules-*}):

  \begin{lstlisting}[language=TypeScript,caption={Public entrypoint import (allowed)},label={lst:g1-do-public-import}]
// DO: import via public entrypoint (handler, not entity)
import { PlaceOrderHandler } from '@tiny-store/modules-orders';
  \end{lstlisting}

  \begin{lstlisting}[language=TypeScript,caption={Cross-module import of internals (blocked)},label={lst:g1-dont-internals-import}]
// DON'T: entities and repositories are internal to the module
import { Order } from '@tiny-store/modules-orders';
import { OrderRepository } from '@tiny-store/modules-orders';
  \end{lstlisting}

  \noindent
  Reactions to order state changes flow through event contracts and listeners; requests use public handlers. Cross-cutting primitives belong in \texttt{libs/shared/}.

  \item \emph{Step 2: Encode Explicit Declarations.}
  Express \emph{requires} and \emph{forbids} in a single, executable source of truth:

  \begin{lstlisting}[language=TypeScript,caption={Code-based boundary descriptor},label={lst:g1-descriptor}]
export const moduleBoundaries = {
  orders: {
    requires: ['shared-domain', 'shared-infrastructure'],
    forbids: ['inventory', 'payments', 'shipments'],
  },
  inventory: {
    requires: ['shared-domain', 'shared-infrastructure'],
    forbids: ['orders', 'payments', 'shipments'],
  },
} as const;
  \end{lstlisting}

  \item \emph{Step 3: Centralize Cross-Module Wiring.}

\begin{lstlisting}[language=TypeScript,caption={Composition root wiring (\texttt{register-listeners.ts})},label={lst:g1-do-wiring}]
// DO: centralized in the composition root (auditable)
import { OrderPlacedListener } from '@tiny-store/modules-inventory';
eventBus.subscribe('OrderPlaced', (e) => new OrderPlacedListener().handle(e));
\end{lstlisting}

\begin{lstlisting}[language=TypeScript,caption={Hidden wiring inside a module (avoid)},label={lst:g1-dont-hidden-wiring}]
// DON'T: wiring inside a module makes dependencies implicit
EventBus.getInstance().subscribe('OrderPlaced', handler);
\end{lstlisting}

  \item \emph{Step 4: Enforce Boundary Verification as a First-Class Gate.}
  The gate is binary: pass (no violations) or fail with exact dependency location.

\begin{lstlisting}[language=bash,caption={Boundary verification gate},label={lst:g1-boundary-gate}]
npm run test:boundary

# Boundary violation detected
# Source:    libs/modules/orders
# Forbidden: libs/modules/inventory/src/domain/Product.ts
# Fix: use public handler, event+listener, or promote to libs/shared/
\end{lstlisting}

\begin{lstlisting}[language=TypeScript,caption={Public surface test},label={lst:g1-boundary-spec}]
describe('Module Public Surface', () => {
  it('does not expose Inventory internals', async () => {
    const inventory = await import('@tiny-store/modules-inventory');
    expect((inventory as any).Product).toBeUndefined();
    expect((inventory as any).InventoryRepository).toBeUndefined();
  });
  it('exposes Orders public handlers', async () => {
    const orders = await import('@tiny-store/modules-orders');
    expect((orders as any).PlaceOrderHandler).toBeDefined();
  });
});
\end{lstlisting}

  \item \emph{Step 5: Add Module Isolation Checks.}
  Detect runtime coupling invisible to static analysis:

  \begin{lstlisting}[language=TypeScript,caption={Module isolation test},label={lst:g1-isolation}]
describe('Orders module isolation', () => {
  it('loads with declared dependencies only', async () => {
    const module = await loadModule('orders', {
      allowed: ['shared-domain', 'shared-infrastructure'],
    });
    expect(module).toBeDefined();
  });
});
  \end{lstlisting}

\end{itemize}

\subsubsection*{Exercises}
\begin{itemize}[itemsep=10pt,topsep=4pt]

  \item \emph{Exercise 0: Confirm the Baseline.}
  Re-run Step 0; all checks pass.

  \item \emph{Exercise 1: Encapsulation Leak (export internal symbols).}
  In \texttt{libs/modules/orders/src/index.ts}:

  \begin{lstlisting}[language=TypeScript,caption={Violation: exporting internal symbols},label={lst:g1-ex1-leak}]
// VIOLATION: exporting internal domain symbols leaks encapsulation
export * from './domain/entities/order';
export * from './domain/repositories/order-repository';
  \end{lstlisting}

  \begin{lstlisting}[language=bash,caption={Run boundary enforcement after Exercise 1},label={lst:g1-ex1-run}]
npm run test:boundary    # FAILS: C_leak increases
  \end{lstlisting}

  \noindent
  \textbf{Fix:} remove these exports; expose a contract instead (handler/port/event).

  \item \emph{Exercise 2: Direct Cross-Module Import (undeclared dependency).}
  In an Orders feature file:

  \begin{lstlisting}[language=TypeScript,caption={Violation: direct cross-module import},label={lst:g1-ex2-import}]
// VIOLATION: Orders directly imports Inventory
import { SomeInventoryHandler } from '@tiny-store/modules-inventory';
  \end{lstlisting}

  \begin{lstlisting}[language=bash,caption={Run boundary enforcement after Exercise 2},label={lst:g1-ex2-run}]
npm run test:boundary    # FAILS: C_undecl or C_forbid increases
  \end{lstlisting}

  \noindent
  \textbf{Fix:} replace with event-driven collaboration (publish \texttt{OrderPlaced}, react in Inventory).

\subsection*{Conclusion}

G1 demonstrated that boundary enforcement in a modular monolith requires executable constraints, not documentation alone. The exercises confirmed that each violation produces a specific, measurable signal while the system remains a single deployable unit. With boundaries enforced (G1), the next concern shifts from \emph{whether} modules can remain isolated to \emph{how} they can be evolved safely, motivating G2.
